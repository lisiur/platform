| Token | Value | Usage |

| xs | 4px | Icon gaps within menu items, inline padding |

| md | 16px | Default element spacing, tree level indentation |
| lg | 24px | Section padding, form field gaps |

| Exception | Value | Rationale |

## Interaction Contracts

### Role-Menu Assignment Page

**Layout:** Left-right split (matching existing menu management pattern from Phase 2).

```
┌─────────────────────────────────────────────────────┐
│  Page Header                                         │
│  "角色菜单分配" + description                         │
├──────────────┬──────────────────────────────────────┤
│              │                                      │
│  Left Panel  │  Right Panel                         │
│  (1/3 width) │  (2/3 width)                        │
│              │                                      │
│  ┌──────────┐│  ┌────────────────────────────────┐  │
│  │ Role     ││  │  Menu Tree with Checkboxes      │  │
│  │ Selector ││  │                                 │  │
│  │ (Select) ││  │  ☑ Dashboard                   │  │
│  └──────────┘│  │  ├─ ☑ Users                    │  │
│              │  │  ├─ ☐ Organizations            │  │
│              │  │  └─ ☑ Roles                    │  │
│              │  │      ├─ ☑ Role Menu Assign     │  │
│              │  │      └─ ☐ Role Permissions     │  │
│              │  │                                 │  │
│              │  │  [保存分配]                      │  │
│              │  └────────────────────────────────┘  │
└──────────────┴──────────────────────────────────────┘
```

**Interaction Flow:**

1. Admin lands on `/roles/[roleId]/menus` (or `/roles/menus` with role selector)
2. Admin selects a role from the dropdown → tree loads with current assignments pre-checked
3. Admin toggles checkboxes on/off → auto-includes children when checking a parent (D-03)
4. Admin clicks "保存分配" → PUT request sent, success/error toast shown
5. Changes take effect on next page load (Zustand store re-fetch)

### Checkbox Tree Behavior

- **Check parent → all descendants checked** (D-03)
- **Uncheck parent → all descendants unchecked** (cascading uncheck)
- **Check child without parent → parent remains unchecked** (no auto-include upward)
- **Visual:** Checkbox component from shadcn (`@/components/ui/checkbox`)
- **Tree structure:** Reuse existing `buildTree()` pattern from `menu-tree.tsx`
- **Icons:** Reuse existing `getIcon()` pattern from `menu-tree.tsx`
- **Expand/collapse:** Reuse existing expand/collapse toggle pattern

### Dynamic Sidebar

**Loading flow:**

1. User logs in → `useMenuStore.fetchMenus()` called on mount
2. While loading: `Skeleton` components render in sidebar (existing `SidebarMenuSkeleton` pattern)
3. Menus arrive → tree built in Zustand store → sidebar renders collapsible tree
4. `bottomMenuItems` (profile, settings) remain hardcoded below the dynamic section

**Sidebar layout (top to bottom):**

```
┌──────────────────────┐
│  [App Logo / Name]   │
├──────────────────────┤
│  Dynamic Menu Tree   │  ← from useMenuStore
│  ├─ ▸ Dashboard      │
│  ├─ ▸ Users          │
│  ├─ ▸ Roles          │
│  │  ├─ Role Menu     │
│  │  └─ Permissions   │
│  └─ ▸ Organizations  │
├──────────────────────┤
│  Profile              │  ← hardcoded bottomMenuItems
│  Settings             │
└──────────────────────┘
```

**States:**

| State | Visual | Trigger |
|-------|--------|---------|
| Loading | Skeleton placeholders in sidebar | `useMenuStore.loading === true` |
| Loaded | Dynamic tree renders | `useMenuStore.fetched === true` |
| Error | Previous menu state retained (silent fallback) | Fetch fails |
| Empty | "No menus available" text + hardcoded bottom items | API returns empty array |
